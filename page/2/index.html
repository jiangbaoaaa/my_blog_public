摘要

本报告走读 Autoware 自由空间规划与局部执行模块的源码实现。核心结论：在 PARKING 场景下，freespace_planner 节点订阅 costmap + route + 当前位姿，经 onTimer 三重判定（场景/数据/完成态）后调用 algo_->makePlan()（freespace_planner_node.cpp:487），其内核是 Hybrid A*（astar_search.cpp）；Hybrid A* 在 (x,y,θ) 三维网格上用运动基元扩展、以 Reeds-Shepp 距离（reeds_shepp.cpp）为启发函数搜出含倒车段的路径，按 is_back 标记分段发布为 Trajectory。轨迹执行阶段，由 surround_obstacle_checker（起步周围防撞，限速 0）与 obstacle_cruise_planner（行驶中前方停车/RSS 巡航）两层兜底，确保验收口径①（轨迹不穿越障碍）。

RS 曲线生成逻辑

RS 曲线在规划框架中的定位

汽车不能像机器人那样原地旋转，必须遵守非完整运动学约束（受限于最小转弯半径）。

  • RS 功能：在最小转弯半径约束下、允许前进 + 倒车，连接两个位姿 (x, y, θ) 的最短路径。是 Dubins 曲线（只能前进）的推广。
  • 为什么泊车要用它：侧方位停车与倒车入库必须包含倒车段，仅支持前进的普通几何曲线无法满足；RS 给出由前进段与倒车段拼接的最短解。
  • 在本模块的作用：RS 是 freespace_planning_algorithms 算法库里的几何工具。RS 曲线并非直接求解从起点到终点的最终路径，而是作为图搜索算法（A* / RRT*）的节点拓展工具，负责两个功能：
    1. 启发函数（heuristic）：估计当前点到终点的剩余距离，引导 A* 向终点搜索（astar_search.cpp:227）。
    2. shoot / 终结：搜索到某节点时以 RS 曲线连接终点，命中即接入（详见第 3 章）。

衔接：Day1 的 makePlan（L487）→ A* → estimateCost → 本节 RS；下游：Day3 Hybrid A* 深读。

核心数学结算：CSC 等生成器内部调用 LpSpLp 等函数，再被核心函数 reedsShepp() 依次调用，每次都比较当前算出的路径长度与已有最短长度，保留最短。

对外计算接口：reedsShepp(s0,s1)，外界获取最短路径的方法——先将起点与终点之间的相对差值进行平移旋转，转化为相对于原点的目标点，再除以最小转弯半径把问题归一化到单位圆的几何问题，调用匿名函数结算最短路径。

轨迹离散化：interpolate 给定一条 RS path 和一个长度 seg（通常是离散步长，如 0.1 m），通过积分算出在该弧长处车辆的实际坐标，使外层 A* 等全局算法每隔 0.1 m 采样一点，再到 costmap 上判断是否发生碰撞。

reedsShepp 取最短入口（cpp L608）—— 整个 RS 生成的总调度

path 是引用传入每个分组函数，初值 length=max。每个分组里每算出一个候选 L，只要 L < path.length() 就覆盖 path。因此取最短并非最后比较一次，而是逐候选贪心更新——5 个分组执行完毕，path 即为全局最短。

RS 曲线生成逻辑

三种形状 × 两种方向 = 6 基元

RS 将任意路径划分为若干段，每段为下列 3 种形状之一，方向由长度的正负决定：

形状 枚举（hpp L97） 曲率 弧长正 弧长负
直行 RS_STRAIGHT 0 前进直行 倒车直行
左转弧 RS_LEFT +1/rho_ 前进左转 倒车左转
右转弧 RS_RIGHT −1/rho_ 前进右转 倒车右转
（占位） RS_NOP — 未用段以此填充 —

→ 3 形状 × 2 方向 = 6 基元（S+/S−/L+/L−/R+/R−）。RS_NOP 仅用于把不足 5 段的路径补齐。

公式函数命名规则：LpSpLp = Left-plus、Straight-plus、Left-plus（全前进的”左-直-左”）；m=minus=倒车，如 LpRmL=左前-右倒-左前。

生成 + 取最短流程

⭐ 关键认知：分段常曲率，G1 连续但曲率不连续

interpolate（cpp L670）逐段行进，每段为恒定曲率的运动：

含义：

  • 每段内部曲率恒定（直=0，弧=±1/rho_）。
  • 段与段交界处曲率跳变（如 0 → 1/rho_，或 +1/rho_ → −1/rho_）。
  • 所以 RS 路径是 G1 连续（位置 + 切向连续，车辆可连续跟踪），但不是 C² 曲率连续。

⚠️ 对接 W3/W4 验收：若验收口径②要求曲率连续，RS 单独不满足（段间曲率阶跃）——需要后接样条/回旋曲线（clothoid）平滑，或由控制器做限幅。

Hybrid A* 搜索

为什么要有 Hybrid A*

普通 A* 的路径在栅格中心之间以折线或 45° 方向连接，真实车辆（阿克曼转向，受最小转弯半径约束）无法跟踪。若直接将 A* 路径交给控制器，往往跟踪失败，或需进行大量后处理（平滑、拟合），且拟合可能失败。

Hybrid A* 的目标：直接在车辆的位姿空间中搜索，使所得路径本身即可被车辆跟踪执行。

A* Hybrid A*
状态 栅格 (i, j) 连续位姿 (x, y, θ)，θ 是车头朝向
邻居生成 上下左右/对角 用车辆运动学前进一步（运动基元 motion primitive）
路径形状 栅格折线 真实可行驶轨迹（圆弧/RS 曲线段拼接）

核心改动：状态从离散栅格 → 连续位姿。 把朝向 θ 也纳入状态，搜索空间变成 (x, y, θ)，节点之间的转移边直接用满足运动学的基元（直行、定半径圆弧）。在本模块中，Hybrid A* 是 makePlan 的内核——Day1 节点层仅负责将 costmap 与起终点传入算法、将输出封装为 Trajectory；而”在地图上搜索一条可通行且支持倒车的路径”由本层完成。Day2 的 RS 曲线在此仅作为启发函数。

Hybrid A* 搜索逻辑

状态空间：(x, y, θ) 三维网格离散

连续位姿没法直接搜索，所以 setMap（L138）按 costmap 分辨率把 x, y 切成网格，再按 theta_size（默认 48）把航向 θ 切成 48 档，得到 3D 数组 nodes_[y][x][theta]，每格放一个 AstarNode。任意连续位姿经 pose2index 吸附到最近格子——这正是 Hybrid A* 控制状态数量、避免无限细分的关键（以离散栅格做去重与记忆，以连续状态做路径生成，即 Hybrid 之义）。gc 连续累积（不量化），但节点身份按格子去重（status 三态），不会无限扩展。

运动基元表：用对称性一次生成

createTransitionTable（L76）离线枚举所有合法的单步行进基元：

基元 生成位置 含义
前进直行 forward_straight（L95） 走 step_min = R_min·dtheta 直线
前进左弧 forward_left（L100） 半径 R 左转 dtheta
前进右弧 forward_left.flipped()（L102） 镜像得右转
倒车各向 .reversed()（L116） 前进各基元取反，得倒车版本

半径从 R_min 到 R_max 分 turning_radius_size+1 档（默认 12 档），每档有左、右两条弧再加一条直行。每个 θ 档把基元用 rotated(theta) 转到当前朝向下（L107–112）；use_back=true 时再追加 reversed() 倒车基元（L114–118）。所有基元弧半径都 ≥ R_min，所以每条转移边天然满足最小转弯半径约束——这是 Hybrid A* 路径可被车辆跟踪的根本保证。

代价与启发

  • gc：从起点到当前节点的累积代价，每步加 move_cost。
  • move_cost：正常就是基元 distance；若这一步换了方向（is_back 翻转，即前进↔倒车切换），乘 reverse_weight（默认 2.0，L265）——此为换向惩罚，使算法倾向于减少换向次数。
  • hc：estimateCost 给的 RS 距离 × distance_heuristic_weight。
  • cost = gc + hc，优先队列每次取最小。

reverse_weight 的存在解释了最终轨迹中倒车段较少、分段不频繁的原因：每次换向需付出 2 倍代价，算法倾向于将路径组织为长前进段与长倒车段，而非反复进退。

终点判定与回溯

isGoal（L336）并非要求精确等于终点，而是检查节点是否落入终点周围的范围盒：横向 < lateral_goal_range、纵向 < longitudinal_goal_range、航向差 < angle_goal_range（都除以 2 作半宽）。落入范围盒即视为到达；only_behind_solutions 还能要求解必须从终点后方进入。

命中后 setPath（L300）从终点节点沿 parent 指针回溯至起点，收集每个节点位姿和 is_back，最后 std::reverse（L328）反转成起点→终点序。这里的 is_back 字段一路传到 Day1 节点层：createTrajectory 根据它给速度取正负号，getReversingIndices 再标出倒车切换点——倒车段的来源即在此步。

局部避障触发机制

为什么需要局部避障

规划轨迹基于规划时刻的地图计算。车辆行驶过程中环境是动态的：行人可能进入轨迹、前车可能减速、障碍物可能突然出现，从而产生碰撞风险。因此规划栈在 freespace_planner 之后设置两层运行时兜底，按速度区间分工：

节点 场景 检测动作 触发动作
surround_obstacle_checker 低速 / 起步 车身周围（360°）最近障碍 距离过近 → 限速 0
obstacle_cruise_planner 行驶中 轨迹前方的障碍 有碰撞风险 → 插停车点 / 限速跟车

surround_obstacle_checker：车身周围有障碍 → 阻止起步

定位

该节点订阅点云与动态物体，将车辆表示为一个多边形（createSelfPolygon，node.cpp L123），用 Boost.Geometry 计算车身到最近障碍的距离。距离过近时发布 max_velocity = 0 的限速指令，阻止车辆起步。内部是两态状态机（node.hpp L58）：

触发距离为 2.0 m、恢复距离为 2.5 m，恢复阈值大于触发阈值，构成滞回（hysteresis），避免障碍位于边界附近时状态频繁切换。
该节点仅在车辆已停止时触发（isStopRequired 中 if (!is_vehicle_stopped) return false;，node.cpp L428）。其职责是防止起步，而非制动行驶中的车辆——行驶中的制动由 obstacle_cruise_planner 负责。

定位与两层规划

行驶中轨迹前方出现障碍时由该节点处理，将障碍分为两类，分别采用两条规划路径：

  • 需停车：障碍完全阻挡，需在障碍前插入零速点 → generateStopTrajectory
  • 需跟车：障碍为运动物体（前车），计算安全跟车限速并减速跟随 → generateCruiseTrajectory

PlannerInterface 是基类（planner_interface.hpp L41），巡航部分是纯虚函数 generateCruiseTrajectory（L72），有两个实现：OptimizationBasedPlanner（node.cpp L351，默认，用优化解速度）和 PIDBasedPlanner（L354，用 PID）。停车逻辑 generateStopTrajectory 写在基类里（L69），两种 planner 共用。

整条触发链在 onTrajectory（node.cpp L533）里串起来：

障碍筛选为目标：getTargetObstacles（L662）对每个感知物体，用 polygon_utils 计算其多边形与自车轨迹的交点（collision_points）。只有轨迹确实穿过障碍（存在 collision_points）的物体才进入 target_obstacles。这一步是整个触发的总开关：无交点即无碰撞风险，直接忽略。

停车触发：generateStopTrajectory（planner_interface.cpp L86）

从 target_obstacles 中选取最近的停车障碍，计算到碰撞点的弧长距离，减去安全 margin 与车头偏移，得到停车位置，再由 insertStopPoint（L196）将零速点插入轨迹。其中包含关键可行性检查 calcMinimumDistanceToStop（L75），即制动距离公式：

$$d_{brake} = \frac{v_0^2}{2,|a_{min}|}$$

含义：以当前车速 $v_0$、采用极限制动 $a_{min}$ 所需的最小停车距离。若障碍比该距离更近（closest_obstacle_stop_dist < feasible_stop_dist，L186），表明即使极限制动也无法避免碰撞，于是置 will_collide_with_obstacle = true 发告警（L184-189）。这是兜底的安全诊断，不是触发条件本身。

巡航触发：generateCruiseTrajectory + RSS 距离

对于需跟车的障碍（前车），采用 RSS（Responsibility-Sensitive Safety）模型计算安全跟车距离，定义在 calcRSSDistance（planner_interface.hpp L127）：

$$d_{rss} = v_{ego}\cdot t_{idle} + \frac{v_{ego}^2}{2,|a_{ego}|} - \frac{v_{obj}^2}{2,|a_{obj}|} + margin$$

四项分别是：自车反应时间走的距离 + 自车制动距离 − 障碍自己的制动距离 + 余量。第三项被减去，是因为前车同时也在减速，无需为其预留完全停下所需的整段距离。计算所得 $d_{rss}$ 用于确定限速：跟车距离不足则降速，充足则维持。输出为 VelocityLimit（限速指令），而非紧急停车——这正是与停车触发的区别。

第九章：自动驾驶车辆的离线地图构建

点云建图的流程

上图是整个建图流程：

  1. 从给定的ROS数据包中解出IMU、RTK和激光雷达数据。使用IMU与激光雷达数据组成LIO系统。
  2. 大部分自动驾驶车辆会携带RTK设备或者组合导航设备。按照RTK或者组合导航的位姿给每个关键帧赋一个RTK位姿，作为观测。
  3. 使用LIO作为相邻帧运动观测，使用RTK位姿作为绝对坐标观测，优化整条轨迹并且判定各关键帧的RTK有效性。这是第一轮优化。
  4. 在上一步的基础上对地图进行回环检测，可以使用简单的基于欧氏距离的检测，使用NDT或者常见的配准算法计算相对位姿。
  5. 再把信息放到统一的位姿图中进行优化，消除累计误差带来的重影，同时移除失效区域。这是第二轮优化。确定位姿之后，可以按照这些位姿到处点云地图。为了方便地图加载和查看，可以对点云进行切片处理。处理之后的点云可以用来进行高精度定位或者地图标注。

前端实现

  1. 将ROS数据包中的RTK数据提取出来，放在RTK消息队列中，按照采集时间排序
  2. 用第一个有效的RTK数据作为地图原点，用其他的RTK读书减去原点，作为RTK位置观测
  3. 用IMU和激光雷达输出数据运行LIO，按照距离和角度阈值抽取关键帧，再保存关键帧结果。抽取关键帧时，也会从LIO中获取它的位姿和点云

后续的建图流程可以从前端的结果出发，不必解析ROS数据包。先使用RTK位姿和LIO位姿进行联合优化，在此基础上进行回环检测，消除累计误差。

后端位姿图优化和异常值检验

我们希望最终优化的轨迹能够考虑到各种传感器的输入，因此后端图优化需要包含以下因子：

  1. RTK因子。这是一种绝对的位姿观测数据。
  2. 雷达里程计信息，使用前面介绍的激光雷达里程计或者LIO作为局部连续性约束
  3. 回环因子，执行回环检测就加载回环检测的相关信息，把它们放入图优化中。

回环检测

点云地图的质量很大程度上取决于回环检测是，设计回环检测的步骤如下：

  1. 遍历第一轮优化轨迹中的关键帧。对每两个在空间上相隔较近，但是在时间上存在一定距离的关键帧进行一次回环检测。
  2. 对于每个关键帧对，使用Scan to Map 的配准方法，在关键帧附近抽取一定数量的点云作为子地图，然后对扫描和子地图进行配准。
  3. 由于每个检查点的计算都是独立的，使用并发编程让第二步能够在机器上并发执行，加快速度
  4. 配准的实际执行由NDT完成，使用NDT的分值作为回环有效性的判断依据。
  5. 记录所有成功的回环，并且在下次调用优化算法时读取回环检测的结果。

地图的导出

多数应用中，希望控制实时点云的载入规模，只加载自身周围200米范围内的点云，其他范围的点云视情况卸载，重新组织本章建立的点云，按照100米的边长进行分块，设计分块加载和卸载的接口。点云的切分实际上是根据每个点的坐标计算其所在的网格，然后将它投到对应的网格中。切分完地图之后，没快递图会独立到处一个PCD文件，程序可以快速读取到地图区块的位置。

激光雷达传感器与点云的数学模型

激光雷达传感器的分类与数学模型

Lidar(Light detection and ranging)即激光雷达，与其他领域Radar(radio detection and ranging）表述的雷达有所不同，其中自动驾驶中还有一种毫米波雷达被称为Radar.

自动驾驶使用的激光雷达主要分为机械旋转式激光雷达和固态激光雷达两种

  1. 机械旋转式激光雷达（Spining Lidar）:可以看成是一列以固定频率旋转的激光探头。每个探头都能够快速探测外部物体离自身的距离。每旋转一周就可以完成一次对周围场景的扫描。可以根据线数进一步细分，常见包括4、8、16、32、64、80、128线等。
  2. 固态激光雷达（Solid State Lidar）：固态激光雷达本身不会进行360°扫描，只能探测约120°视野范围的3D信息，与RGBD相机十分相似。

扫描方式 整体宏观机械旋转 微观镜片偏转 (MEMS) 或 无机械运动 (OPA/Flash)

视场角 单个可实现360°水平覆盖 单个视场角有限（如120°），需多个拼接
可靠性/寿命 较低（有磨损，不耐振动） 很高（无/微运动部件，抗振性好）
成本 非常高 量产后有潜力变得很低
体积/重量 大、重 小、轻，易于集成
技术成熟度 非常成熟 发展中，MEMS相对成熟
主要应用 自动驾驶原型车、机器人、高精度地图测绘 大规模量产的乘用车、工业自动化、消费电子

最近邻问题

在实际应用中，不止关心最近的一个点，而是更关心以下两种扩展形式

  1. k-最近邻（K-Nearest Neighbors）：找到距离查询点最近的k个点，例如找到距离最近的5个点
  2. 半径最近邻（Radius Neighbors Search）:找到距离查询点某个半径r 范围内所有的特征点，如找到发哦周围100米 内所有邻居

一个点的特性必须通过分析它和周围邻近点的关系体现，因此，最近邻搜索几乎是所有点云处理算法的基石和第一步，有很多处理思路

暴力最近邻

暴力最近邻（Brute-force Nearest Neighbour Search）是最简单直观的最近邻搜索。遍历每一个点，找到最近的那一个，至于暴力K近邻则是找到最近的K个。

栅格与体素方法

栅格核心思想： 将一个2D平面分割成一个由许多个大小相同的小方块（单元格或像素） 组成的规则网格。每个小方块（Cell）都有一个唯一的坐标（如行号和列号），并被赋予一个值来表示该区域的某种属性。

体素核心思想：将一个3D空间分割成一个由许多个大小相同的小立方体（体素） 组成的规则网格。每个小立方体就是一个体素，它也有一个唯一的3D坐标（如X, Y, Z索引），并被赋予一个值来表示该3D空间区域的属性。

拟合问题

在传统SLAM应用中，关心的元素是基本的、静态的，更关心如何使用这些元素帮助我们进行点云之间的匹配和配准。在配准问题中，常见的做法是先利用最近邻结构找到一个点的若干最近邻，再对这些最近邻进行拟合。

平面拟合

简单来说，就是从一个三维空间中的点云集合中，找出一个“最理想”的平面，使得这个平面能够最好代表这些点的整体分布趋势，而最理想通常意味着所有点到这个平面的总距离之和最小。

解决平面拟合问题最主流的方法：

  1. 最小二乘：适用于当点云数据相对干净，大部分点都确实分布在一个平面附近，只有少量高斯噪声的情况下。这个问题可以使用的技术是主成分分析（Principal Component Analysis, PCA）

  1. RANSAC（Random Sample Consensus 随机采样一致性）：

在实际应用中，通常会将两者结合：

  1. 首先使用RANSAC从包含各种物体和噪声的原始点云中，稳健识别属于某个平台的内点集合

  2. 然后，对于只包含内点的集合应用PCA，精确计算出最终平面方程。

这样利用了RANSAC鲁棒性，又利用最小二乘的精确性。

直线拟合

与平面拟合一样，仍然可以使用RANSAC和最小二乘法来拟合直线，使用参数方程来描述直线，而用一个一般方程描述平面。

第八章：紧耦合LIO系统

紧耦合的原理和优点

紧耦合是一种传感器融合策略，不再是融合结果，而是融合“原始数据”或者“原始约束”

优势：

  1. 极高的精度：松耦合在第一步计算里程计时，已经丢弃了原始点云信息，只保留了一个位姿估计，紧耦合利用了所有的原始信息，IMU与LiDAR双向校正
  2. 更强的鲁棒性：当LiDAR 失效或者是IMU剧烈抖动时，可以受到其他传感器的约束，系统能够输出稳定、平滑的位姿。

基于IEKF的LIO系统

IEKF状态变量与运动方程

在紧耦合LIO系统中，IMU与点云配准使用同样的状态模型、运动方程与观测方程。

标准的EKF通过在当前状态估计点对非线性系统进行一次泰勒展开，然后应用标准卡尔曼滤波。但是对于LIO这种强非线性系统，一次线性化会引入巨大误差。IEKF是对EKF的重大改进，在更新步骤中，不只是线性化一次，而是进行多次迭代。每次迭代中，都围绕上一次迭代更新后的最新状态点重新进行线性化，允许IEKF逐步逼近非线性测量函数的真实最优解。

实现基于IEKF的LIO系统

紧耦合LIO系统的点云残差可以有多种计算方式：如点到线、点到面、NDT，等等。本次实现一个基于IEKF的NDT LIO，实现基于点到面ICP的LIO。

  1. 预积分因子通常关联两个关键帧的高维状态。在转换为图优化问题时，可以选择把各个顶点分开处理。

第六章：2D SLAM

2D SLAM 的基本原理

相比较于三维空间的点云，2D SLAM可以视为在俯视视角下工作的激光SLAM算法。在这种视角下，激光雷达扫描数据和地图数据可以被简化为二维形式。但是任然还有难以解决的问题，例如：大部分2D SLAM方案预设障碍物与激光雷达传感器在同一高度，如果存在其他高度的障碍物，或者物品形状随着高度有明显变化，那么二维地图就无法有效表达这类物体，导致机器人发生碰撞。本次将以现代的视角展示2D SLAM，现在展示一下典型的2D SLAM 框架。

流程

  1. 数据输入：二维激光雷达传感器按照一定的频率向外输出激光测距的信息，产生一系列点云，这就是“扫描数据”

  2. 前端：扫描匹配。为了计算扫描对应的机器位姿，需要将它与某些东西进行配准，称为扫描匹配。既可以是将两次扫描的结果进行匹配，也可以是将它与地图进行匹配，所以扫描匹配可以分为Scan to Scan 和 Scan to Map 两种模式。扫描匹配的结果会被用来持续更新机器人的位姿，这个位姿输出就是实时的定位结果。

  3. 前端：插入子地图。当估计这个扫描的位姿之后，将这帧数据插入到当前子地图中，是的子地图不断增长。

    子地图： 这是一个关键概念，系统不是把所有的点塞入一张大地图中，而是线构建一系列小的、内部一致的子地图，当一个子地图累计足够数据之后，就会被封存为历史子地图。

  4. 前端：SLAM输出。系统根据地图中的点云数据，实时更新一个占据栅格地图，这就是最终的地图输出。占据栅格地图会区分地图中的障碍物还有可通行区域，可以过滤掉运动物体对于地图的影响。

  5. 后端：扫描数据+历史子地图–> 回环检测。当一个新的扫描数据产生时，回环检测模块会将其与所有历史子地图进行比较，以此来判断机器人是否回到一个以前来过的地方

  6. 后端：全局优化。如果回环检测成功，就会在位姿图中添加一个回环约束，系统会出发一次全局优化。优化器会调整所有历史位姿与子地图的位置，是的所有约束误差最小。

似然场法的核心思想就是，不只是在占据栅格地图上的栅格给高分，还要在已占据的栅格附近都给不错的分数。总结来说就是通过预先将地图模糊化，将一个尖锐的匹配问题，转化为一个平滑的问题，极大提高了2D SLAM 中定位和建图的鲁棒性。

占据栅格地图的原理

占据栅格地图的核心就是：将地图划分为无数个均匀小方格，为每一个小方格赋予一个概率值，这个值代表这个小方格被障碍物占据的可能性的代销。

在最简单的模型中，每个格子只有三种状态：

  1. 占据：有障碍物
  2. 空闲：格子是空的，可以通行
  3. 未知：不知道格子里是什么

由于栅格地图本身存在分辨率限制，将连续的激光线条转换成栅格地图中每个格子的概率增减时，存在一个栅格化过程，如下右图所示，在二维栅格地图中，有两种栅格化方法：

  1. 选择对每个激光线条计算栅格化结果
  2. 选择先计算一个固定大小的模版区域

这就引出了两种地图生成的方法

这是一种对直线进行栅格化的算法，通常用于几何直线矢量化。可以完全由整数运算实现，适用于刷新栅格地图。

基于模版的地图生成

如果不使用直线填充算法，可以考虑使用模版算法进行栅格填充。模版区域内每个格子的角度和距离都是预先算好的，需要更新栅格地图时，将模版中每隔格子的距离值与激光在该角度下的距离值进行比较。如果末班中的距离小于激光打到的距离，就可以认为该格子是空白的。如果等于激光打到的距离值，则认为是占据的。如果大于，则该格子不做更新。

子地图的原理

把匹配算法和栅格地图放在一起，生成一个匹配好的结果，将若干个匹配好的结果放在一起，就形成了一个子地图。子地图是介于单个扫描数据和全局地图之间的数据形式，它把若干个扫描数据按照时间顺序归到一起，使用起来比较灵活。

按照下面的逻辑实现一个基于子地图的2D SLAM

  1. 一个子地图对应一个似然场和一个栅格地图
  2. 总是把当前的扫描数据与当前的子地图进行匹配，得到的扫描数据在当前子地图中的位姿
  3. 如果机器人发生位移或者旋转，则按照一定距离和角度选取关键帧
  4. 如果机器人的移动范围超过了当前的子地图，或者当前子地图包含的关键帧超出一定数量，就建立一个新的子地图，新的子地图以当前帧为中心
  5. 合并每一个子地图的栅格地图，就可以得到全局地图

这就是2D SLAM 的建图过程

回环检测与闭环

当机器人走完一圈回到原来的场地时，新的子地图会与原来的子地图存在一定重叠区域，如果不对这些重叠区域做任何配准，那就会在拼接的时候出现重影，只要把当前的扫描数据或者子地图与历史地图配准，再调整各子地图之间的相对位置关系，就可以消除累计误差了。但是仍然有小问题需要处理：

  1. 应该检查哪些历史子地图？
  2. 检测回环时用到的配准方法与里程计中的配准方法稍有不同，里程计的配准是基于连续运动的，回环的位姿初始值是由累计误差的多少决定的。于是，实际的回环配准算法通常是在原有的ICP或者似然场基础上，增加一些针对较差位姿初始值的处理方式，例如网格搜索、粒子滤波、分枝定界、金字塔法。
  3. 在检测到回环之后，需要对整个地图进行位姿修正，既可以基于关键帧的位姿来做，也可以基于子地图的位姿来做。虽然使用子地图的优化问题规模会小很多，但是无法修复单个子地图内部的畸变，因此现实中也有很多系统仍然使用关键帧作为基本的数据单元。

回环检测并不是一步完成的，分为多个阶段：

  1. 阶段一：候选帧搜索（位置识别）
  2. 阶段二：候选帧验证（几何校正）
  3. 阶段三：回环修正（图优化）

阶段一：候选帧搜索

可以使用视觉SLAM中的词袋模型来实挑选候选帧，这个同样适合激光SLAM。也可以使用扫描上下文的方式

阶段二：候选帧验证

使用不同的配准方法将当前帧与候选帧进行配准，验证匹配得分，如果满足条件则确认找到了回环

阶段三：回环修正

既然已经有了一个高精度的约束，那么可以修正整个SLAM轨迹和地图，使得图的总误差最小。

多分辨率的回环检测

金字塔式的回环检测被称为由粗至精的，或者多分辨率的配准方法。或者多分辨率的配准方法。分枝定界法既是一种用于粗配准的方法，也是整数规划中的一个重要方法。至于多分辨率的似然场匹配，依然是一个扫描匹配问题，为了减少初始值不准确带来的影响，可以在原来的似然场之外，增加一些其他分辨率的似然场。可以在小分辨率的似然场中先进行一次匹配，然后把粗配准的结果投影到高分辨率的似然场中，星城由粗至精的匹配过程。

系统会为每一个级别的地图单独生成一个高斯似然场，从粗到精，先在最高层（80cm），这个层是最模糊、最平滑的似然场进行扫描匹配。会得到一个粗略的位姿估计，再接着在中间层（40cm ）将上一步得到的粗略位姿作为新的、更准的位姿，进一步修整，得到更为精确的位姿，最后在原始高精度的似然场上进行最后一次精修，得到最终的高精度位姿。

虽然机器人可能回到了原来的位置，但是可能由于朝向不同，导致匹配度不够，所以在多分辨率配准中，只要匹配点数超过所有的扫描数据的一定比例，就可以认为地图被匹配上了。这个阈值表示回环检测的敏感程度。

基于子地图的回环修正

如果当前帧与子地图之间存在配准关系，就会启动一次回环修正。

通过多分辨率匹配计算了当前帧在历史子地图中的位姿，在回环优化中，以每隔子地图的位姿为优化变量，构建一个位置图进行优化。位置图的观测主要来源是：一种是相邻位姿图之间的相对位姿观测，另一种是回环检测计算出来的两个子地图之间的相对位姿关系。为了防止检测到错误的回环，可以添加一个回环的验证过程，如果修正的累计误差太大，则回环被当成异常值剔除。

激光雷达的运动补偿、反光等问题

底盘与激光雷达在同时旋转的时候，会出现相对运动，这个过程会影响激光雷达每个点的实际测量距离，这个过程被称为运动畸变。可以由运动补偿算法去除。

无法处理回环检测（Loop Closure）。当系统回到原点时，滤波方法无法利用该信息来修正整个历史轨迹的累积误差。 这是图优化的王牌优势。一个回环检测可以在图中两个相距很远的节点间建立一条强约束边。优化器会调整整条轨迹来满足这个约束，从而显著消除全局漂移，获得全局一致的地图和轨迹。

IMU系统的运动学

激光SLAM

问题：

  1. 频率低
  2. 运动畸变
  3. 特征稀疏场景失效

惯性导航

IMU通常包含加速度计和陀螺仪，可以高频测量设备自身的三轴加速度和三轴角速度。

作用：

  1. 高频位姿估计
  2. 运动畸变校正
  3. 为激光匹配提供初始值

缺点：

IMU是通过对加速度和角速度进行积分来推算位姿的，这个过程存在累计误差。时间一长，哪怕是很小的噪声也会被不断放大，导致推算出的位置和姿态会像“脱缰的野马”一样飘走（即漂移），无法长期独立工作。

组合导航

组合导航就是将激光SLAM和惯性导航组合起来，形成一个激光惯性SLAM系统。核心思想：

用IMU高频、短时精确的特性，弥补激光雷达的低频和畸变问题；同时，用激光SLAM长期、全局精确的特性，来不断校正IMU的累计误差。

技术上主要分为两种融合方式：

  1. 松耦合：这是一种比较简单直接的融合方式。激光SLAM和IMU各自独立计算出自己的位姿，然后通过一个滤波器（如卡尔曼滤波器）将这两个“结果”进行融合。它实现简单，但因为没有利用原始数据，精度和鲁棒性相对较差。
  2. 紧耦合：这是一种更先进、更主流的融合方式。它将IMU的测量值（角速度、加速度）和激光雷达的原始特征（点、线、面）共同放入一个大的优化问题中进行求解（通常使用因子图优化或复杂的卡尔曼滤波器）。在这种框架下，激光的观测可以非常直接地校正IMU的状态（如零偏），IMU的数据也可以更精确地指导点云处理。紧耦合虽然复杂，但性能更优，鲁棒性更强，是当前研究和应用的热点。

这样就得到了纯粹的运动加速度，如果对速度进行一次时间积分，就得到了速度，对速度进行再次时间积分，就得到了位置。

需要注意的是，要把IMU放在车辆中心，否则可能会在车辆发生旋转和移动的时候，测量到由车辆旋转导致的离心力、科氏力和角加速度。会影响IMU的读数。

IMU测量方程中的噪声模型

IMU噪声由两部分组成：测量噪声和零偏。在实际卡尔曼滤波算法中，系统状态不仅包括姿态、位置、速度，还会把偏置作为一个状态量来进行在线估计和校正，抑制IMU的漂移。噪声是滤波器的重要输入，通常通过艾伦方差工具对IMU进行离线标定获得。

使用IMU进行航迹推算

在已知此时刻位置速度以及姿态的前提下，通过测量IMU数据，获得角速度以及加速度，来更新姿态以及位置速度。

航迹推算是一个迭代积分的过程，在每个时间步，先用陀螺仪积分更新姿态，再利用新姿态消除加速度计读数中的重力，最后对纯运动加速度进行二次积分来更新速度和位置。在实际系统中，必须与其他传感器（GPS、LiDAR、相机等）进行融合，由这些外部传感器来不断校正IMU的累积误差，从而实现长期、鲁棒的定位。

卫星导航

我们日常生活中最熟悉的“GPS”其实是美国开发的系统的名称，而这类系统的通用术语是GNSS（Global Navigation Satellite System，全球卫星导航系统）。卫星导航实际上可以提供车辆所需的所有定位信息，包括车辆的位置、姿态、速度等物理量。

卫星定位的最终精度受很多因素影响，人们提出各种校正技术消除已知误差。PPP（Precise Point Positioning 精密单点定位）、**RTK(Real-Time Kinematic 实时动态差分)**等技术取得进展，使得定位精度不断提升。

差分全球卫星导航系统中差分技术是消除误差最核心，最有效的方法之一，基本思想是“通过比较消除共同误差”

主要技术

  • SBAS (星基增强系统)：这是一种广域差分系统。它由多个地面站组成网络，计算出大范围（如整个大陆）的修正信息，然后通过地球同步卫星广播给用户。例如美国的 WAAS、欧洲的 EGNOS，以及覆盖日本的 MSAS。
  • RTK (实时动态) 技术：这是一种更高精度的载波相位差分技术。它不仅使用差分修正伪距，还利用了信号的载波相位进行测量。通过解算整周模糊度，RTK技术可以实现厘米级的实时定位精度，是测绘、精准农业和自动驾驶等领域的核心技术。

两个蘑菇头就是RTK接收器，通常使用两个接收器组成双天线方案，根据两根天线的位置差计算车辆实时朝向。

常见的世界坐标系

地球上最常见的坐标系就是经纬度坐标系，即地理坐标系。而UTM（Universsal Transverse Mercator Grid System）坐标系是将地球视为一个椭球体，投影至横躺圆柱体上，将其展开并且进行分区得到的。

使用UTM的优点是使用了米制单位，与其他传感器兼容性好，缺点是某些地区可能存在两个分区的跨界处，需要进行额外的坐标处理。

使用误差状态卡尔曼滤波器实现组合导航

RTK设备为我们提供一个不太稳定的位姿观测源，可以使用RTK与IMU结合，使用拓展卡尔曼滤波器形成组合导航算法。严格来说，是使用误差状态卡尔曼滤波器（Error-State Kalman Filter, ESKF）。

误差状态卡尔曼滤波器的核心思想是：不直接估计整个状态，而是将状态分为两部分：一个大的“标称状态”和一个小的“误差状态”。

  1. 标称状态 (x): 使用IMU的原始测量值，通过非线性运动学方程进行高频积分。这个状态包含我们主要关心的量：位置 p、速度 v、姿态 q（通常用四元数表示）。
  2. 误差状态 (δx): 使用一个线性的卡尔曼滤波器来估计标称状态中存在的微小误差。这个误差状态通常包括：位置误差 δp、速度误差 δv、姿态误差 δθ（一个小的旋转矢量）、以及IMU的偏置误差 ba,bg.

整个过程是一个经典的“预测-校正”循环。

速度观测量

在GINS（GNSS/INS）系统中，长时间缺少RTK数据，ESKF就变为纯靠IMU积分的递推模式。该模式下位移将快速发散， 因为缺少速度观测。大部分轮式机器人会携带一个编码器测定自身速度，进而推算自身的局部运动情况。这些速度观测与IMU结合，可以形成局部的航迹推算，可以简单将他们融入ESKF，作为速度观测量。

轮式编码器它被安装在车轮的转轴或电机上，用于测量车轮的旋转运动。通过测量轮子转了多少圈、转了多快以及朝哪个方向转，我们就可以间接计算出机器人或车辆的行驶距离、速度和方向。

其特点可以概括为：

  1. 单个轮式编码器只输出轮子转过的距离，可以对这个距离进行采样。如果每隔固定时间采样，就可以看成对速度的测量值。
  2. 如果本身在底盘上的角度是固定不动的，可以通过编码器计算车辆位移
  3. 轮子本身智能测量车辆在前进方向上的速度位移，无法测量平行方向上或者高度的速度位移
  4. 实际中，车轮容易打滑，轮子可能会发生空转，导致编码器虽然有输出，胆识车辆本身并没有前进的状态。
