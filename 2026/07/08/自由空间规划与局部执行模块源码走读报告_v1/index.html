摘要

本报告走读 Autoware 自由空间规划与局部执行模块的源码实现。核心结论：在 PARKING 场景下，freespace_planner 节点订阅 costmap + route + 当前位姿，经 onTimer 三重判定（场景/数据/完成态）后调用 algo_->makePlan()（freespace_planner_node.cpp:487），其内核是 Hybrid A*（astar_search.cpp）；Hybrid A* 在 (x,y,θ) 三维网格上用运动基元扩展、以 Reeds-Shepp 距离（reeds_shepp.cpp）为启发函数搜出含倒车段的路径，按 is_back 标记分段发布为 Trajectory。轨迹执行阶段，由 surround_obstacle_checker（起步周围防撞，限速 0）与 obstacle_cruise_planner（行驶中前方停车/RSS 巡航）两层兜底，确保验收口径①（轨迹不穿越障碍）。

freespace_planner 节点：输入→算法→输出调用链

节点定位

  • 节点名：/planning/scenario_planning/parking/freespace_planner
  • 激活时机：上游 scenario_selector 切到 PARKING 场景时。
    • 切换条件 = 车当前位姿在 parking_lot 多边形内（scenario_selector_node.cpp:158 selectScenarioByPosition()，isInParkingLot 查的是 current_pose，第 162 行）。
  • 节点职责：在 PARKING 场景下，把 车位终点位姿（goal）转换为含倒车段的 Trajectory（velocity 可为负），供控制器跟随。
  • **算法可替换：默认 astar（Hybrid A*），可选 rrtstar（RRT*），由参数 planning_algorithm 决定。

ROS1→ROS2：节点没有可见的 while(spin) 主循环，由订阅/发布与定时器回调驱动。ROS2 节点的关键组成依次为：构造 → 订阅 → 发布 → 定时器 → 核心调用。

RS 曲线生成逻辑

RS 曲线在规划框架中的定位

汽车不能像机器人那样原地旋转，必须遵守非完整运动学约束（受限于最小转弯半径）。

  • RS 功能：在最小转弯半径约束下、允许前进 + 倒车，连接两个位姿 (x, y, θ) 的最短路径。是 Dubins 曲线（只能前进）的推广。
  • 为什么泊车要用它：侧方位停车与倒车入库必须包含倒车段，仅支持前进的普通几何曲线无法满足；RS 给出由前进段与倒车段拼接的最短解。
  • 在本模块的作用：RS 是 freespace_planning_algorithms 算法库里的几何工具。RS 曲线并非直接求解从起点到终点的最终路径，而是作为图搜索算法（A* / RRT*）的节点拓展工具，负责两个功能：
    1. 启发函数（heuristic）：估计当前点到终点的剩余距离，引导 A* 向终点搜索（astar_search.cpp:227）。
    2. shoot / 终结：搜索到某节点时以 RS 曲线连接终点，命中即接入（详见第 3 章）。

衔接：Day1 的 makePlan（L487）→ A* → estimateCost → 本节 RS；下游：Day3 Hybrid A* 深读。

核心数学结算：CSC 等生成器内部调用 LpSpLp 等函数，再被核心函数 reedsShepp() 依次调用，每次都比较当前算出的路径长度与已有最短长度，保留最短。

对外计算接口：reedsShepp(s0,s1)，外界获取最短路径的方法——先将起点与终点之间的相对差值进行平移旋转，转化为相对于原点的目标点，再除以最小转弯半径把问题归一化到单位圆的几何问题，调用匿名函数结算最短路径。

轨迹离散化：interpolate 给定一条 RS path 和一个长度 seg（通常是离散步长，如 0.1 m），通过积分算出在该弧长处车辆的实际坐标，使外层 A* 等全局算法每隔 0.1 m 采样一点，再到 costmap 上判断是否发生碰撞。

reedsShepp 取最短入口（cpp L608）—— 整个 RS 生成的总调度

path 是引用传入每个分组函数，初值 length=max。每个分组里每算出一个候选 L，只要 L < path.length() 就覆盖 path。因此取最短并非最后比较一次，而是逐候选贪心更新——5 个分组执行完毕，path 即为全局最短。

CSC 分组生成器（cpp L158）—— 对称变换翻倍的样板

distance（cpp L656）—— A* 启发函数最终调的就是它

这就是 Day1→Day2 的接口：A* 的 estimateCost（astar_search.cpp:227）→ calcReedsSheppDistance（L31）→ 本函数 → 为 A* 提供到终点剩余距离的启发值。

RS 曲线生成逻辑

三种形状 × 两种方向 = 6 基元

RS 将任意路径划分为若干段，每段为下列 3 种形状之一，方向由长度的正负决定：

形状 枚举（hpp L97） 曲率 弧长正 弧长负
直行 RS_STRAIGHT 0 前进直行 倒车直行
左转弧 RS_LEFT +1/rho_ 前进左转 倒车左转
右转弧 RS_RIGHT −1/rho_ 前进右转 倒车右转
（占位） RS_NOP — 未用段以此填充 —

→ 3 形状 × 2 方向 = 6 基元（S+/S−/L+/L−/R+/R−）。RS_NOP 仅用于把不足 5 段的路径补齐。

公式函数命名规则：LpSpLp = Left-plus、Straight-plus、Left-plus（全前进的”左-直-左”）；m=minus=倒车，如 LpRmL=左前-右倒-左前。

含义：

  • 每段内部曲率恒定（直=0，弧=±1/rho_）。
  • 段与段交界处曲率跳变（如 0 → 1/rho_，或 +1/rho_ → −1/rho_）。

Hybrid A* 搜索

为什么要有 Hybrid A*

普通 A* 的路径在栅格中心之间以折线或 45° 方向连接，真实车辆（阿克曼转向，受最小转弯半径约束）无法跟踪。若直接将 A* 路径交给控制器，往往跟踪失败，或需进行大量后处理（平滑、拟合），且拟合可能失败。

Hybrid A* 的目标：直接在车辆的位姿空间中搜索，使所得路径本身即可被车辆跟踪执行。

A* Hybrid A*
状态 栅格 (i, j) 连续位姿 (x, y, θ)，θ 是车头朝向
邻居生成 上下左右/对角 用车辆运动学前进一步（运动基元 motion primitive）
路径形状 栅格折线 真实可行驶轨迹（圆弧/RS 曲线段拼接）

核心改动：状态从离散栅格 → 连续位姿。 把朝向 θ 也纳入状态，搜索空间变成 (x, y, θ)，节点之间的转移边直接用满足运动学的基元（直行、定半径圆弧）。在本模块中，Hybrid A* 是 makePlan 的内核——Day1 节点层仅负责将 costmap 与起终点传入算法、将输出封装为 Trajectory；而”在地图上搜索一条可通行且支持倒车的路径”由本层完成。Day2 的 RS 曲线在此仅作为启发函数。

Hybrid A* 搜索逻辑

状态空间：(x, y, θ) 三维网格离散

连续位姿没法直接搜索，所以 setMap（L138）按 costmap 分辨率把 x, y 切成网格，再按 theta_size（默认 48）把航向 θ 切成 48 档，得到 3D 数组 nodes_[y][x][theta]，每格放一个 AstarNode。任意连续位姿经 pose2index 吸附到最近格子——这正是 Hybrid A* 控制状态数量、避免无限细分的关键（以离散栅格做去重与记忆，以连续状态做路径生成，即 Hybrid 之义）。gc 连续累积（不量化），但节点身份按格子去重（status 三态），不会无限扩展。

运动基元表：用对称性一次生成

createTransitionTable（L76）离线枚举所有合法的单步行进基元：

基元 生成位置 含义
前进直行 forward_straight（L95） 走 step_min = R_min·dtheta 直线
前进左弧 forward_left（L100） 半径 R 左转 dtheta
前进右弧 forward_left.flipped()（L102） 镜像得右转
倒车各向 .reversed()（L116） 前进各基元取反，得倒车版本

半径从 R_min 到 R_max 分 turning_radius_size+1 档（默认 12 档），每档有左、右两条弧再加一条直行。每个 θ 档把基元用 rotated(theta) 转到当前朝向下（L107–112）；use_back=true 时再追加 reversed() 倒车基元（L114–118）。所有基元弧半径都 ≥ R_min，所以每条转移边天然满足最小转弯半径约束——这是 Hybrid A* 路径可被车辆跟踪的根本保证。

代价与启发

  • gc：从起点到当前节点的累积代价，每步加 move_cost。
  • move_cost：正常就是基元 distance；若这一步换了方向（is_back 翻转，即前进↔倒车切换），乘 reverse_weight（默认 2.0，L265）——此为换向惩罚，使算法倾向于减少换向次数。
  • hc：estimateCost 给的 RS 距离 × distance_heuristic_weight。
  • cost = gc + hc，优先队列每次取最小。

reverse_weight 的存在解释了最终轨迹中倒车段较少、分段不频繁的原因：每次换向需付出 2 倍代价，算法倾向于将路径组织为长前进段与长倒车段，而非反复进退。

终点判定与回溯

isGoal（L336）并非要求精确等于终点，而是检查节点是否落入终点周围的范围盒：横向 < lateral_goal_range、纵向 < longitudinal_goal_range、航向差 < angle_goal_range（都除以 2 作半宽）。落入范围盒即视为到达；only_behind_solutions 还能要求解必须从终点后方进入。

命中后 setPath（L300）从终点节点沿 parent 指针回溯至起点，收集每个节点位姿和 is_back，最后 std::reverse（L328）反转成起点→终点序。这里的 is_back 字段一路传到 Day1 节点层：createTrajectory 根据它给速度取正负号，getReversingIndices 再标出倒车切换点——倒车段的来源即在此步。

局部避障触发机制

为什么需要局部避障

规划轨迹基于规划时刻的地图计算。车辆行驶过程中环境是动态的：行人可能进入轨迹、前车可能减速、障碍物可能突然出现，从而产生碰撞风险。因此规划栈在 freespace_planner 之后设置两层运行时兜底，按速度区间分工：

节点 场景 检测动作 触发动作
surround_obstacle_checker 低速 / 起步 车身周围（360°）最近障碍 距离过近 → 限速 0
obstacle_cruise_planner 行驶中 轨迹前方的障碍 有碰撞风险 → 插停车点 / 限速跟车

surround_obstacle_checker：车身周围有障碍 → 阻止起步

定位

该节点订阅点云与动态物体，将车辆表示为一个多边形（createSelfPolygon，node.cpp L123），用 Boost.Geometry 计算车身到最近障碍的距离。距离过近时发布 max_velocity = 0 的限速指令，阻止车辆起步。内部是两态状态机（node.hpp L58）：

触发距离为 2.0 m、恢复距离为 2.5 m，恢复阈值大于触发阈值，构成滞回（hysteresis），避免障碍位于边界附近时状态频繁切换。
该节点仅在车辆已停止时触发（isStopRequired 中 if (!is_vehicle_stopped) return false;，node.cpp L428）。其职责是防止起步，而非制动行驶中的车辆——行驶中的制动由 obstacle_cruise_planner 负责。

obstacle_cruise_planner：前方障碍 → 停车 或 减速跟车

定位与两层规划

行驶中轨迹前方出现障碍时由该节点处理，将障碍分为两类，分别采用两条规划路径：

  • 需停车：障碍完全阻挡，需在障碍前插入零速点 → generateStopTrajectory
  • 需跟车：障碍为运动物体（前车），计算安全跟车限速并减速跟随 → generateCruiseTrajectory

PlannerInterface 是基类（planner_interface.hpp L41），巡航部分是纯虚函数 generateCruiseTrajectory（L72），有两个实现：OptimizationBasedPlanner（node.cpp L351，默认，用优化解速度）和 PIDBasedPlanner（L354，用 PID）。停车逻辑 generateStopTrajectory 写在基类里（L69），两种 planner 共用。

整条触发链在 onTrajectory（node.cpp L533）里串起来：

障碍筛选为目标：getTargetObstacles（L662）对每个感知物体，用 polygon_utils 计算其多边形与自车轨迹的交点（collision_points）。只有轨迹确实穿过障碍（存在 collision_points）的物体才进入 target_obstacles。这一步是整个触发的总开关：无交点即无碰撞风险，直接忽略。

停车触发：generateStopTrajectory（planner_interface.cpp L86）

从 target_obstacles 中选取最近的停车障碍，计算到碰撞点的弧长距离，减去安全 margin 与车头偏移，得到停车位置，再由 insertStopPoint（L196）将零速点插入轨迹。其中包含关键可行性检查 calcMinimumDistanceToStop（L75），即制动距离公式：

$$d_{brake} = \frac{v_0^2}{2,|a_{min}|}$$

含义：以当前车速 $v_0$、采用极限制动 $a_{min}$ 所需的最小停车距离。若障碍比该距离更近（closest_obstacle_stop_dist < feasible_stop_dist，L186），表明即使极限制动也无法避免碰撞，于是置 will_collide_with_obstacle = true 发告警（L184-189）。这是兜底的安全诊断，不是触发条件本身。

巡航触发：generateCruiseTrajectory + RSS 距离

对于需跟车的障碍（前车），采用 RSS（Responsibility-Sensitive Safety）模型计算安全跟车距离，定义在 calcRSSDistance（planner_interface.hpp L127）：

$$d_{rss} = v_{ego}\cdot t_{idle} + \frac{v_{ego}^2}{2,|a_{ego}|} - \frac{v_{obj}^2}{2,|a_{obj}|} + margin$$

四项分别是：自车反应时间走的距离 + 自车制动距离 − 障碍自己的制动距离 + 余量。第三项被减去，是因为前车同时也在减速，无需为其预留完全停下所需的整段距离。计算所得 $d_{rss}$ 用于确定限速：跟车距离不足则降速，充足则维持。输出为 VelocityLimit（限速指令），而非紧急停车——这正是与停车触发的区别。
