局部避障触发机制

发表于 2026-07-08 分类于 Autoware学习

W2 Day4 · 局部避障触发机制

本文对应报告第 4 节，走读规划栈运行时的两层避障兜底。Day4 负责的交付物是局部避障触发机制：检测到障碍后，满足什么条件才触发停车或减速。

源码：

  • surround_obstacle_checker/src/node.cpp + node.hpp
  • obstacle_cruise_planner/src/node.cpp + planner_interface.cpp + common_structs.hpp

衔接：Day1–3 的 freespace_planner 产出一条理想轨迹；Day4 这两个节点在轨迹跑起来后做”遇障兜底”，对应验收口径①（轨迹不穿越障碍）。

为什么需要局部避障

Day1–3 产出的轨迹是基于规划时刻的地图算出来的。车开起来后环境是动态的：行人可能走进轨迹、前车可能减速、点云里突然冒出一根柱子。如果死守原轨迹，迟早撞。所以规划栈在 freespace_planner 之后挂了两层运行时兜底，分工按”速度区间”切：

节点 管什么场景 检测什么 触发动作
surround_obstacle_checker ==低速/起步== 车身周围（360°）最近障碍 离太近 → 限速 0，不让动
obstacle_cruise_planner ==行驶中== 轨迹前方的障碍 撞得上 → 插停车点 / 限速跟车

一个管”别乱起步”，一个管”开起来别撞”。下面分开走读。

surround_obstacle_checker：车身周围有障碍 → 阻止起步

定位

这个节点订阅点云和动态物体，把车当成一个多边形（createSelfPolygon，node.cpp L123），用 Boost.Geometry 算车身到最近障碍的距离。距离太近就发一个 max_velocity = 0 的限速指令，让车不要起步。

这个节点只在车已经停着的时候才触发（isStopRequired 里 if (!is_vehicle_stopped) return false;，node.cpp L428）。它的角色是”防止起步”，不是”刹停一辆开着的车”——行驶中的制动归下面那个 obstacle_cruise_planner。

触发逻辑

距离怎么来的：getNearestObstacle（L305）分别用点云（L334）和动态物体（L367）各算一个最近距离，取更近的那个。每条路都是 bg::distance(ego_polygon, obstacle)（L356 / L398）——车多边形到障碍的几何最短距离。

状态机在 onTimer（L196，每 100ms 跑一次）里推进，核心两段：

isStopRequired（L425）三段逻辑：车没停 → 不触发；发现障碍 → 触发并记下时间；障碍没了但在 state_clear_time（默认 2.0s）内 → 继续保持 STOP。最后这条是==时间维度的防抖==，配合距离滞回，双重去抖。

关键参数（node.cpp L148–156）

参数 默认值 含义
surround_check_distance 2.0 触发距离(m)，车身到障碍 < 此值 → 限速 0
surround_check_recover_distance 2.5 恢复距离(m)，> 此值才解除（> 触发值，滞回）
state_clear_time 2.0 障碍消失后保持 STOP 的时长(s)
use_pointcloud / use_dynamic_object true 用点云 / 动态物体哪个来源检测

obstacle_cruise_planner：前方障碍 → 停车 或 减速跟车

定位与两层规划

行驶中，轨迹前方有障碍时由它处理。它把障碍分成两类，分别走两条规划路径：

  • 要停的：障碍挡死，必须在障碍前插一个零速点 → generateStopTrajectory
  • 要跟的：障碍在动（前车），算一个安全跟车限速，减速跟着 → generateCruiseTrajectory

PlannerInterface 是基类（planner_interface.hpp L41），巡航部分是纯虚函数 generateCruiseTrajectory（L72），有两个实现：OptimizationBasedPlanner（node.cpp L351，默认，用优化解速度）和 PIDBasedPlanner（L354，用 PID）。停车逻辑 generateStopTrajectory 写在基类里（L69），两种 planner 共用。

整条触发链在 onTrajectory（node.cpp L533）里串起来：

障碍怎么变成 target： getTargetObstacles（L662）对每个感知物体，用 polygon_utils 算它的多边形和自车轨迹的交点（collision_points）。只有==轨迹真的会穿过障碍==（有 collision_points）的物体才进 target_obstacles。这一步是整个触发的总开关：没交点 = 撞不上 = 直接忽略。

停车触发：generateStopTrajectory（planner_interface.cpp L86）

从 target_obstacles 里挑最近的停车障碍，算到碰撞点的弧长距离，减掉安全 margin 和车头偏移，得到该停的位置，再用 insertStopPoint（L196）把一个零速点插进轨迹。

里面有个关键的可行性检查 calcMinimumDistanceToStop（L75）——就是高中刹车距离公式：

$$d_{brake} = \frac{v_0^2}{2,|a_{min}|}$$

含义：以当前车速 $v_0$、用极限制动 $a_{min}$，最少要多少距离才能停下。如果障碍比这个距离还近（closest_obstacle_stop_dist < feasible_stop_dist，L186），说明==按极限制动也停不住==，会撞，于是置 will_collide_with_obstacle = true 发告警（L184-189）。这是个兜底的安全诊断，不是触发条件本身。

巡航触发：generateCruiseTrajectory + RSS 距离

对于要”跟”的障碍（前车），用 RSS（Responsibility-Sensitive Safety）模型算安全跟车距离，定义在 calcRSSDistance（planner_interface.hpp L127）：

$$d_{rss} = v_{ego}\cdot t_{idle} + \frac{v_{ego}^2}{2,|a_{ego}|} - \frac{v_{obj}^2}{2,|a_{obj}|} + margin$$

四项分别是：自车反应时间走的距离 + 自车制动距离 − 障碍自己的制动距离 + 余量。第三项减掉是因为前车也在减速，不需要给它留”完全停下”的整段距离。算出来的 $d_{rss}$ 用来定限速：跟车距离不够就降速，够就维持。输出是 VelocityLimit（限速指令），不是急停——和停车触发的区别就在这。
