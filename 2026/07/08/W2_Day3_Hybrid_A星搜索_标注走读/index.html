Hybrid A*（混合 A*）

为什么要有 Hybrid A*

普通 A* 的路径长这样：在栅格中心之间走折线、走 45°。一辆真实的车（阿克曼转向，有最小转弯半径）根本走不出来。如果直接拿 A* 路径给控制器，要么跟踪不上，要么得做大量后处理（平滑、拟合），还可能拟合失败。

Hybrid A* 的目标：直接在车辆的”位姿空间”里搜索，让搜出来的路径本身就能被车开出来。

核心改动：状态从离散栅格 → 连续位姿

A* Hybrid A*
状态 栅格 (i, j) 连续位姿 (x, y, θ)，θ 是车头朝向
邻居生成 上下左右/对角 用车辆运动学前进一步（运动基元 motion primitive）
路径形状 栅格折线 真实可行驶轨迹（圆弧/RS 曲线段拼接）

每次扩展就是：从当前位姿，分别走这 6 段，得到 6 个候选新位姿。

连续空间怎么搜索？——3D 栅格去重

纯连续位姿有无限多个，没法搜。Hybrid A* 的巧思：

  • 把连续位姿 (x, y, θ) 投影到一个 3D 离散栅格（x 分箱、y 分箱、θ 分箱）。
  • 每个 3D cell 里只保留一个状态（通常是代价最低的那个，或第一个到达的）。
  • 这样搜索空间是有限的（分辨率完备），但每个状态本身又是连续的真实位姿——路径连续平滑，但搜索规模可控。

一句话：用离散栅格做”去重/记忆”，用连续状态做”路径生成”。这就是 “Hybrid（混合）” 二字的含义——离散 + 连续的混合。

启发函数要升级

普通欧氏距离在 Hybrid A* 里太弱（因为还要考虑朝向和不能横移），扩展会爆炸。常用更强的启发：

  1. 2D A* / Dijkstra 距离：先在 2D（忽略 θ、允许任意方向移动）上跑一遍，拿到每个格子到目标的距离，作为 h。这已经比欧氏强很多。
  2. Reeds-Shepp 距离：直接算”如果没障碍，从当前位姿到目标的最短 RS 曲线长度”。更强，但要配合碰撞检查裁剪。

工程上常把两者结合，甚至取多种启发的最大值，兼顾引导力和可采纳性。

Reeds-Shepp 加速：这是 “A* RS” 的 RS 部分

光靠运动基元一格一格扩，最后可能永远”差一点点”对不上目标的精确朝向 θ。解决办法：

  • 每次扩展出一个节点后，尝试用 Reeds-Shepp 曲线直接连到目标。
  • 如果这条 RS 曲线无碰撞，那就直接接上、结束搜索。
  • 这让算法能精确收敛到目标位姿，大幅减少靠近目标时的无谓扩展。

Reeds-Shepp 曲线：给定两个位姿，在”车能前进、能后退”的前提下，存在解析的最短路径（由若干段圆弧+直线组合，公式可查表）。能前进能后退 = 适合泊车这种需要揉库的场景。
Dubins 曲线：只能前进不能后退的对应物，适合单向行驶。

优缺点小结

优点 缺点
路径直接可执行，无需平滑后处理 3D 搜索，比 2D A* 慢
考虑朝向和运动学，适合真实车辆 分辨率完备而非真正完备（栅格分得不够细可能漏解）
RS 射靶能精确收敛到目标位姿 代价/启发调参比 A* 复杂

Hybrid A* 在规划框架中的定位与功能

经典 A* 在二维栅格 (x, y) 上搜索，相邻节点是上下左右四个格子。但这种路径对汽车没有意义——汽车受最小转弯半径约束，不能原地横移、不能任意转向。Hybrid A* 的解决办法是把朝向 θ 也纳入状态，搜索空间变成 (x, y, θ)，节点之间的转移边直接用满足运动学的基元（直行、定半径圆弧）。

在本模块里，Hybrid A* 是 makePlan 真正的内核。Day1 的节点层只负责把 costmap 和起终点喂进来、把输出包成 Trajectory；真正”在地图上找一条能走通、还能倒车的路径”是这里做的。Day2 的 RS 曲线在这里只担任启发函数——估计当前节点到终点还差多远，引导搜索朝终点走，避免盲目扩散。

关键类：AstarSearch 与 AstarNode

Hybrid A* 的核心数据结构有两层：一个是节点 AstarNode（搜索空间里的一个状态），一个是搜索器 AstarSearch（持有网格、待扩展队列、运动基元表）。

节点结构 AstarNode（hpp L48–60）

成员 含义
x / y / theta 节点的位姿（连续值，离散化后落入网格格子）
gc 从起点到当前节点的实际累计代价
hc 启发代价（到当前节点到终点的估计，h cost）
cost() gc + hc（L59），优先队列按它排序
is_back 当前行驶方向，true=倒车（倒车段的来源）
parent 父节点指针，用于搜索成功后从终点回溯到起点
status None / Open / Closed（hpp L36），节点状态

NodeComparison（hpp L62–68）是优先队列的比较器，lhs->cost() > rhs->cost() 让 priority_queue 变成小顶堆——每次取 cost 最小的节点扩展，这正是 A* 的贪心本质。

搜索器 AstarSearch（hpp L105–157）

成员 类型 作用
transition_table_ vector<vector<NodeUpdate>> 运动基元表：每个 θ 档对应一组可走基元
nodes_ vector<vector<vector<AstarNode>>> 3D 网格 [y][x][theta]，状态离散化的载体
openlist_ priority_queue<AstarNode*, …, NodeComparison> 待扩展队列（按 cost 排序）
astar_param_ AstarParam only_behind_solutions / use_back / distance_heuristic_weight
use_reeds_shepp_ bool 是否用 RS 当启发（构造里写死 true，L130）
goal_node_ AstarNode* 命中的终点节点

运动基元 NodeUpdate（hpp L70–103）

一个基元描述”车走一步”的位移量：shift_x / shift_y / shift_theta / distance / is_curve / is_back。它带三个对称变换方法，是生成完整基元表的关键：

  • rotated(theta)（L79）：把基元旋转到当前朝向 θ 下
  • flipped()（L87）：左右镜像（左转变右转）
  • reversed()（L95）：前进变倒车（shift_x 取反、is_back 翻转）

函数 行号 职责

下面贴主循环 search 和启发函数 estimateCost 的带标注代码，这两个是 Day3 的核心。

search 主循环（L235）

主循环的逻辑用一句话讲：反复取”目前看起来最有希望（cost 最小）”的节点，用运动基元往外扩一格，撞墙的扔掉、新的算上代价丢回队列，直到某个节点落进终点范围。 其中 cost = 已经走了多远（gc）+ 估计还剩多远（hc，RS 给的）。

estimateCost 启发函数（L218）——Day2 的 RS 在这里接入

calcReedsSheppDistance（L31）就是 Day2 那套：起一个 ReedsSheppStateSpace(radius)，调它的 distance() 拿到两点间 RS 最短距离。RS 距离之所以是好启发，是因为它本身就是满足运动学的最短路径长度（忽略障碍），比直线距离更贴近车的真实可走距离，能把搜索注意力有效拉向终点。

一个走读发现，写报告时注意区分：本实现里 RS 只当启发函数用（这一处），search 里没有“用 RS 曲线直连终点”的 analytical expansion（shoot）步骤——终点判定靠下面的 isGoal 范围盒。这和教科书 Hybrid A*（带 RS shoot）略有不同，是这份代码的简化。

Hybrid A* 搜索逻辑

状态空间：(x, y, θ) 三维网格离散

连续的位姿 (x, y, θ) 没法直接搜索，所以 setMap（L138）按 costmap 分辨率把 x, y 切成网格，再按 theta_size（默认 48）把航向 θ 切成 48 档，得到一个 3D 数组 nodes_[y][x][theta]。每个格子放一个 AstarNode。任意连续位姿都被 pose2index 吸附到最近的格子——这就是 Hybrid A* 控制状态数量、避免无限细分的关键。

代价的 gc 是连续累积的（不量化），但节点身份是按格子去重的：同一个格子只算一个节点（status 三态），所以不会无限扩展。

运动基元表：用对称性一次生成

createTransitionTable（L76）离线把所有合法的”走一步”枚举出来：

基元 生成位置 含义
前进直行 forward_straight（L95） 走 step_min = R_min·dtheta 直线
前进左弧 forward_left（L100） 半径 R 左转 dtheta
前进右弧 forward_left.flipped()（L102） 镜像得右转
倒车各向 .reversed()（L116） 前进各基元取反，得倒车版本

半径从 R_min 到 R_max 分 turning_radius_size+1 档（默认 12 档），每档有左、右两条弧，再加一条直行。每个 θ 档把这些基元用 rotated(theta) 转到当前朝向下（L107–112）；use_back=true 时再追加 reversed() 的倒车基元（L114–118）。

所有基元的弧半径都 ≥ R_min，所以每条转移边天然满足最小转弯半径约束——这是 Hybrid A* 路径”车能跟”的根本保证。

‘节点拓展演示图’

代价与启发

  • gc：从起点到当前节点的累积代价，每步加 move_cost。
  • move_cost：正常就是基元的 distance；如果这一步换了方向（is_back 翻转，即前进↔倒车切换），乘 reverse_weight（默认 2.0，L265）——这就是换向惩罚，让算法优先少换挡。
  • hc：estimateCost 给的 RS 距离 × distance_heuristic_weight。
  • cost = gc + hc，优先队列每次取最小。

reverse_weight 的存在，解释了为什么最终轨迹的”倒车段”是分段、不频繁的：换一次方向要付 2 倍代价，算法会尽量凑成”前进一长段 + 倒车一长段”，而不是反复进退。

终点判定与回溯

isGoal（L336）不是”恰好等于终点”，而是检查节点是否落进终点周围的范围盒：横向 < lateral_goal_range、纵向 < longitudinal_goal_range、航向差 < angle_goal_range（都除以 2 作半宽）。落进盒子就算到达。only_behind_solutions 还能要求解必须从终点后方进入。

命中后 setPath（L300）从终点节点沿 parent 指针一路回溯到起点，收集每个节点的位姿和 is_back，最后 std::reverse（L328）反转成起点→终点顺序。这里的 is_back 字段会一路传到 Day1 的节点层：createTrajectory 根据它给速度取正负号，getReversingIndices 再标出倒车切换点——倒车段的”源头”就在这一步。

与上下游的衔接

把三天的链路接起来，RS 和 A* 各自的位置就清楚了：

  • RS（Day2） 在 estimateCost(L227) 当启发，给 A* 指”终点在哪”。代表底层数学与运动学工具库。
  • A（Day3）* 在网格上搜出一条满足运动学、避障碍的离散路径，每个点带 is_back。代表A*寻路流程。
  • 节点层（Day1） 把这条路径包成 Trajectory，按 is_back 决定速度正负、分段发布。代表顶层的ROS节点调度与轨迹后处理。
