摘要

本报告在 Autoware planning_simulator 上测试侧方位停车。核心链路：车当前位姿落入 parking_lot 多边形内 → scenario_selector 切 PARKING（scenario_selector_node.cpp:158，isInParkingLot 查 current_pose）→ freespace_planner 经 onTimer 三重判定调用 algo_->makePlan()（freespace_planner_node.cpp:487），内核 Hybrid A* 以 RS 距离为启发搜出含倒车段的路径，按 is_back 标记分段发布为 Trajectory。通过逐参数调整 freespace_planner.param（重点是 use_back、minimum_turning_radius、reverse_weight、goal_range），在临空港地图的 parking_lot A 上跑通侧方入库，并按验收①②③量化轨迹质量。

PARKING 场景触发条件

触发链路定位

侧方停车由 freespace_planner 在 PARKING 场景下完成。场景是否激活由上游 scenario_selector 决定，判定依据来自源码（W2 走读）：

scenario_selector_node.cpp:158 的 selectScenarioByPosition()，其中 isInParkingLot（第 162 行）查的是==车当前位姿 current_pose 是否落在 parking_lot 多边形内==——不是看 goal 在不在车位里。车在多边形内 → 切 PARKING；否则保持 LANE_DRIVING，freespace_planner 不出泊车轨迹。

节点激活后的执行链

PARKING 激活后，freespace_planner 的 onTimer（freespace_planner_node.cpp:414）经三重判定——输入齐全（L417）、当前场景为 PARKING（isActive，L421）、需要重规划（isPlanRequired，L438）——进入 planTrajectory（L468），调用 algo_->makePlan(current, goal)（L487）。makePlan 内核为 Hybrid A*：在 (x,y,θ) 三维网格上用运动基元扩展、以 RS 距离为启发搜出含倒车段的路径，按 is_back 标记分段发布为 Trajectory（倒车段 velocity<0）。

验证话题：/planning/scenario_planning/scenario（应显示 PARKING）与 /planning/scenario_planning/parking/trajectory（应有数据）。

freespace_planner 调参

参数分三类（节点 / 规划通用 / A*），都在 freespace_planner.param.yaml。改完需重编：colcon build --packages-select tier4_planning_launch。

调参对象（速查）

节点级配置

参数 数值 作用
planning_algorithm "astar" 算法选择，泊车用 astar（Hybrid A*）；rrtstar 泊车不稳
waypoints_velocity 1.0 输出轨迹恒速 [m/s]
vehicle_shape_margin_m 0.5 车体碰撞余量 [m]。调大→离障碍远但可通行空间变小易搜不出路；调小→贴边危险。
th_arrived_distance_m 0.5 “到达终点”距离阈值 [m]，影响停车判定
th_course_out_distance_m 1.0 “偏离轨迹”阈值 [m]，超过可能触发重规划

规划通用配置（所有算法共用）

参数 数值 作用
time_limit 30000.0 单次规划超时 [ms]，总搜不出路就调大（当前 30s 已很宽）
max_turning_ratio 0.5 规划最大转向 = 车实际最大转向 × 此比例。 0.5 → 最小转弯半径 ≈ 7.65m。调大(如 0.7)→允许更小半径更灵活但逼近极限。
theta_size 120 整圈角度离散数。越大越平滑但搜索越慢；轨迹锯齿调大到 144
lateral_goal_range 0.5 到达 goal 横向容差 [m]。收紧→停得准但难搜。
longitudinal_goal_range 1.0 到达 goal 纵向容差 [m]
angle_goal_range 6.0 到达 goal 航向容差 [deg]。
curve_weight 0.8 转弯代价。调大→”能直就直”；调小→更短但弯多
reverse_weight 0.7 倒车代价。侧方需要倒车，不能太大；倒车段过多再调大
direction_change_weight 2.0 换向（前进↔倒车）代价。调大→减少来回揉库次数
obstacle_threshold 100 costmap cell ≥ 多少算障碍 [0-255]。空地搜不出路→调高；漏障→调低

A* 配置（planning_algorithm=astar 时生效）

参数 数值 作用
astar.use_back true 是否允许倒车。
astar.search_method "forward" 起步方向：forward（前进起步）/ backward（倒车起步）
astar.adapt_expansion_distance true 近 goal 时用 Reeds-Shepp 解析扩展，大幅加速终点收敛
astar.near_goal_distance 3.0 距 goal 多少 [m] 内尝试 RS 直连。调大→更早连结(快但可能次优)
astar.distance_heuristic_weight 2.0 启发权重（朝 goal 贪心程度）。调大→搜得快但可能次优
astar.smoothness_weight 0.5 平滑度权重。调大→轨迹更平滑。
astar.obstacle_distance_weight 1.75 离障碍距离权重。调大→轨迹远离障碍更安全。
astar.goal_lat_distance_weight 5.0 横向贴近 goal 权重。调大→末点横向更准

仿真结果（侧方停车）

验收①：轨迹不穿越车位边界 / 障碍、形状合理

下面分从前、从后方进入侧方停车位置，均未超过停车区域

验收②：数值校验

测试按”建场景 → 触发 PARKING → 取泊车轨迹 → 数值校验”四步：前两步在 planning_simulator + RViz 手动操作——起点用 2D Pose Estimate 点进 parking_lot 多边形内，再用 2D Goal Pose 点进车位；后两步靠话题订阅与离线脚本。

取数： 离线脚本 ~/traj_eval.py 订阅 /planning/scenario_planning/parking/trajectory（freespace 在 PARKING 下输出，含每个点的 pose 与 longitudinal_velocity_mps，后者 <0 为倒车段）。三项指标计算如下：

指标 计算方法 判定标准
最小转弯半径 $r_{\min}$ 三点 Menger 外接圆 $\kappa=1/r$，取所有三点组半径最小值（直线段跳过） $\geq$ 真车 ==4.5m==（硬约束）
曲率连续性 相邻三点组曲率差 $\max\lvert\Delta\kappa\rvert$ 越小越平滑
末点停位误差 末点与 goal 的位置 $\sqrt{\Delta x^2+\Delta y^2}$、航向 $\lvert\Delta\psi\rvert$ 参考 ≤0.5m / ≤5°

半径口径：==验收标准按真车 4.5m==（硬约束）；当前 max_turning_ratio=0.5 使规划搜索的最小半径 ≈7.65m，大于 4.5m，满足。实测值以 traj_eval.py 算出的 $r_{\min}$ 为准。

验收③：理想跟随到终点

simple_planning_simulator 理想跟随含倒车段的泊车轨迹，车从起点运动并停在车位终点。验收①的前进 / 后退入库动图（见上）即跟随过程的直接证据——车按规划轨迹完成侧方入库并停住；末点是否落在车位终点、是否停稳，以验收②的末点停位误差数据为据（参考 ≤0.5m / ≤5°）。

结论与总结

三条验收达标判定

验收 标准 实测
① 轨迹不穿越车位边界 / 障碍、形状合理 不穿 A 边界、呈”前进 + 倒车”侧方形态 前进 / 后退两种入位均未越界（见验收①动图）
② 曲率连续、最小转弯半径满足、末点停位误差达标 $r_{\min}\geq$ 4.5m、$\max\lvert\Delta\kappa\rvert$ 小、末点 ≤0.5m / ≤5° 见Fig 16
③ 理想跟随跑通到终点 simple_planning_simulator 跟到车位终点 车按轨迹入库停住（见验收①动图 + 验收②末点误差）

链路跑通结论

侧方停车 PARKING 链路在临空港地图 parking_lot A 上==跑通==：起点落入 parking_lot 多边形 → scenario_selector 切 PARKING（:158 / :162）→ freespace_planner makePlan（:487）输出含倒车段轨迹 → 车侧方入库。

调参核心结论

最终采用配置见”基线配置”（关键项：astar.use_back=true、max_turning_ratio=0.5 → 半径 ≈7.65m、theta_size=120）。三条关键发现：

  • ==astar.use_back 必须为 true==，否则侧方倒车段无解。
  • theta_size（70→120）配合 astar.smoothness_weight（0.5→1.0），轨迹更平滑、波动更小（见调参记录 #2）。
  • 收紧 angle / lateral / longitudinal_goal_range 并加大 astar.goal_lat_distance_weight（5.0→7.0），末点更贴近 goal（见调参记录 #3）。
