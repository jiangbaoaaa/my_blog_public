高层次架构

规划组块包括很多子组件

  • Mission Planning：这个模块利用地图数据计算从当前位置到目的地的路线，类似于汽车导航路线规划

  • Planning Modules：这些模块为指定任务规划车辆的行为，包括：目标轨迹、闪烁信号。分为行为与运动类别：

    • Behavior:专注于计算安全且符合规则要求的路线，管理车道变更、交叉路口输入以及停靠站的决策
    • Motion：与行为模块配合使用，确定车辆的行驶轨迹，同时考虑其运动和骑行舒适性，包括路线形成和速度计算的横向与纵向规划

  • Validation：确保计划轨迹的安全性和适当性，具备应急响应能力，在计划轨迹不合适的情况下，会触发应急协议货生成替代路径

自定义功能

下图展示多种结合外部功能的方法

  1. 向规划部件(planning component)添加新模块

可以使用新模块来增强或者替换现有的规划功能，允许在所需的操作车技领域（ODDs）添加功能ODD,或者简化现有功能

  1. 替换规划的子组件

遵循规划组件中的内部接口在此级别进行协作和拓展是可行的

  1. 更换整个规划模块

开发自动驾驶规划系统的组织或研究机构可能有意将其专有的规划解决方案与Autoware的感知或控制模块集成。可以通过更换整个规划系统,遵循组件之间定义的稳健且稳定的接口来实现。但需要注意的是,与现有规划模块的直接协调可能无法实现。

组件接口

本节介绍规划组成部分还有内部模块的输入输出，详细参考规划模块接口：https://autowarefoundation.github.io/autoware-documentation/main/design/autoware-architecture-v1/interfaces/components/planning/#input

输入：

  1. 地图：

    • 矢量图(Vector map)：包含所有关于环境的静态信息，包括用于规划的车道信息，用于生成参考路径的车道几何信息以及交通规则相关信息

  2. 感知：

    • 发现物体信息：提供有关无法提前知晓的物体的实时信息,例如行人和其他车辆。规划组件计划操作,以避免与这些物体发生碰撞。

    • 障碍物信息：提供有关障碍物位置的实时信息,该信息比被探测物体更原始,用于紧急停车和其他安全措施。

    • 占据栅格地图信息：提供有关行人及其他车辆存在的实时信息以及区域信息。

    • 交通信号灯识别信息：实时提供每个交通信号灯的当前状态。规划组件提取规划路径的相关信息,并确定是否在交叉路口停留。

  3. 定位:

    • 车辆运动信息:包括自我车辆的位置、速度、加速度及其他与运动相关的数据

  4. 系统：

    • 操作模式：表明车辆是否处于自主模式

  5. 人机界面(HMI)：

    • 功能执行：允许人工操作员执行或者授权自动驾驶操作，例如车道变更或进入交叉路口

  6. API：

    • 目标(Goal):代表规划组成部分的最终位置
    • Checkpoint:代表通往目的地途中的中点。此操作用于路线计算
    • Velocity limit:为车辆设定最高限速

输出

  1. 控制：

    • 轨迹：提供控制组件必须遵循的平滑姿势、扭动和加速序列。轨迹通常为10秒,分辨率为0.1秒
    • 转弯信号：根据计划操作控制车辆的转弯指示器,例如右转、左、危险等

  2. 系统：

    • 诊断:报告规划组件的状态,说明处理是否正常运行,以及是否正在生成安全计划

  3. HMI：

    • 功能执行可用性:表示可执行或需要的操作状态,例如车道变更或进入交叉路口
    • 轨迹候选:显示用户执行后执行的潜在轨迹

  4. API层：

    • 规划因子:提供有关当前规划行为背后原因的信息。这可能包括目标对象的位置、导致决定停止的障碍以及其他相关信息

规划部分的内部接口

  1. 任务规划到情景规划(Mission Planning to Scenario Planning)

    核心数据：路线(Route)

    作用：提供从起点到终点的宏观指导。你可以把它理解为手机导航软件规划出的整体路线，主要基于高精地图上的车道 ID (Lane IDs) 等信息生成。

    特点：这是一个比较粗略的层级。它只告诉你“应该走哪几条路”，但不会明确指示车辆具体应该行驶在特定车道的左侧还是右侧，一条 Route 中可能包含多条并行的可用车道。

  2. 行为规划到运动规划 (Behavior Planning to Motion Planning)

    核心数据 1： 路径 (Path)。提供车辆需要遵循的粗略位置和速度信息。与宏观的路线不同，路径更为具体，路径点（Path points）通常以大约 1米 的空间间隔分布。如果改变这个默认间隔，可能会直接影响规划组件的计算精度或系统性能。

    核心数据 2： 可行驶区域 (Drivable area)。划定了车辆在物理上可以安全行驶的绝对边界（例如车道线内或无障碍物区域）。下层的运动规划器必须以此为硬性约束，确保最终计算出的实际行驶轨迹严格限制在这个安全多边形范围内。

  3. 场景规划到验证模块 (Scenario Planning to Validation)

    核心数据： 轨迹 (Trajectory)

    作用解释： 明确给出底层控制组件（Control Component）需要努力去追踪的精确指令，包含了期望的位置、速度和加速度。

    特点： 与按空间距离划分的“路径 (Path)”不同，“轨迹 (Trajectory)”通常是按时间划分的。为了满足动力学要求，轨迹点通常以大约 0.1秒 的时间间隔密集定义。

  4. 验证模块到控制组件 (Validation to Control Component)

    核心数据： 带有安全保障的最终轨迹 (Trajectory)

    作用解释： 数据格式与上一步骤中的轨迹完全相同，但在正式下发给控制组件（去实际操作方向盘、油门和刹车）之前，必须经过 Validation（验证模块）的过滤与检查。这一步会加入额外的安全考量（例如防碰撞冗余检测、动力学极限校验），确保交给车辆底盘的指令是绝对安全可执行的。

当前架构中的重要信息

引入场景规划层（Introducing the Scenario Planning layer）

  • 核心痛点：结构化道路 vs. 非结构化空间

    车辆在规则的马路上（有明确的车道线、红绿灯）行驶，与在空旷的停车场（自由空间 Free-space）里行驶，所需的规划逻辑是完全不同的。在马路上，规划器可以依赖高精地图的车道 ID 进行引导；但在停车场，这种基于车道 ID 的方法就失效了。

  • 解决方案：按场景切换

    Autoware 引入了一个机制，在“场景”级别进行子模块切换。当车辆在路上时，启用“车道行驶 (Lane Driving)”模块；进入停车场后，切换为“泊车 (Parking)”模块。

行为与运动的分离 (Separation of Behavior and Motion)

  • Behavior（行为规划）：负责做决策。比如：决定超车、决定避让、决定路口停止。
  • Motion（运动规划）：负责具体的执行动作。比如：计算出精确的转向角、加速度，生成平滑的物理轨迹。

如果两者完全分离（目前做法）：

弊端是“过于保守”：由于行为模块在下达“超车”指令时，并不知道底层的运动模块最终能否算出一条完美的物理轨迹。为了避免发生危险，行为模块往往会做出非常保守的决策（比如：稍微觉得空间不够，就干脆放弃超车）。这会降低车辆的整体行驶效率和表现。

如果两或者融合在一起：

弊端是“扩展性极差”：如果决策和运动计算强行绑定，它们就会相互依赖。假设你现在想为某个特定地区新增一条当地的交通规则（这本应只是 Behavior 层面的逻辑修改），由于代码高度耦合，你可能不得不连带着修改底层的 Motion 轨迹算法代码，导致系统极难维护和扩展。

自定义当前实现中的功能

以下介绍一个当前模块级别拓展方法的说明

在行为路径规划器或者行为速度规划器中添加新模块

像 behavior_path_planner 和 behavior_velocity_planner 这类 ROS 节点，提供了基于插件 (plugins) 的模块接口，通过遵循这些接口规范来添加模块，可以实现模块的动态加载和卸载 (dynamic loading/unloading)。

在规划组件中添加一个新的ROS节点

如果要在运动规划 (Motion Planning) 中添加模块，必须将其创建为一个独立的 ROS 节点并整合到规划组件中。目前的配置方式是：接收上游计算出的目标轨迹，向其中添加信息。在此过程中引入新的 ROS 节点即可实现功能的扩展。

添加或者替换场景

当前的实现引入了场景级别的切换逻辑，这是一种可以集体切换多个模块的方法。这允许添加全新的场景（例如：高速公路驾驶）。

具体做法是：将新场景创建为一个 ROS 节点，并将其与 scenario_selector（场景选择器）ROS 节点对齐以完成集成。

优点：引入重大的新功能时，不会影响其他现有场景（如“车道行驶”）的代码实现。

缺点：只能通过切换场景来进行场景级别的协调，无法在现有的底层规划模块级别进行细粒度的协调。

仓库结构

具体仓库结构：

重要仓库

算法功能核心库（包含规划与控制代码）

包含感知、定位、规划、控制等实际运行算法

  • autoware_core：核心稳定库。这里存放的是高质量、经过充分测试的基础自动驾驶依赖包。由 Autoware 基金会官方维护，主打稳定可靠。
  • autoware_universe：前沿扩展库。这里存放的是实验性质的、最前沿的自动驾驶包。由基金会和开源社区共同维护。大部分新的功能特性和算法迭代都在这里进行。

消息接口定义库（通信标准）

定义不同模块之间的数据结构

  • autoware_msgs：组件间通信消息。比如“感知组件”把障碍物信息发给“规划组件”，用的就是这里定义的数据格式。仿真软件与系统交互也用它。
  • autoware_internal_msgs：组件内通信消息。比如规划组件内部，路线模块和轨迹生成模块之间的数据传递格式。
  • autoware_adapi_msgs：外部 API 通信消息。用于 Autoware 系统与外界（如车辆 HMI 界面、远程操作平台、车队管理系统）交互的标准接口。

启动与配置库

  • autoware_launch：存放 Autoware 的启动文件（launch files）。无论你是想跑规划仿真（planning simulation）、端到端仿真，还是回放数据包（rosbag-replay），都需要通过这里的配置一键启动整个系统。

工具与基础通用库

  • autoware_tools：开发辅助工具。包含了很多用于诊断和调试的工具，这些工具在实车自动驾驶运行时不需要启动，主要是给开发者修 Bug 用的。
  • autoware_utils：通用基础库。包含很多底层的数学计算（math）、几何处理（geometry）和系统工具包，供上层算法调用以避免重复造轮子。

摘要

本报告在 Autoware planning_simulator 上测试侧方位停车。核心链路：车当前位姿落入 parking_lot 多边形内 → scenario_selector 切 PARKING（scenario_selector_node.cpp:158，isInParkingLot 查 current_pose）→ freespace_planner 经 onTimer 三重判定调用 algo_->makePlan()（freespace_planner_node.cpp:487），内核 Hybrid A* 以 RS 距离为启发搜出含倒车段的路径，按 is_back 标记分段发布为 Trajectory。通过逐参数调整 freespace_planner.param（重点是 use_back、minimum_turning_radius、reverse_weight、goal_range），在临空港地图的 parking_lot A 上跑通侧方入库，并按验收①②③量化轨迹质量。

验收②：数值校验

测试按”建场景 → 触发 PARKING → 取泊车轨迹 → 数值校验”四步：前两步在 planning_simulator + RViz 手动操作——起点用 2D Pose Estimate 点进 parking_lot 多边形内，再用 2D Goal Pose 点进车位；后两步靠话题订阅与离线脚本。

取数： 离线脚本 ~/traj_eval.py 订阅 /planning/scenario_planning/parking/trajectory（freespace 在 PARKING 下输出，含每个点的 pose 与 longitudinal_velocity_mps，后者 <0 为倒车段）。三项指标计算如下：

指标 计算方法 判定标准
最小转弯半径 $r_{\min}$ 三点 Menger 外接圆 $\kappa=1/r$，取所有三点组半径最小值（直线段跳过） $\geq$ 真车 ==4.5m==（硬约束）
曲率连续性 相邻三点组曲率差 $\max\lvert\Delta\kappa\rvert$ 越小越平滑
末点停位误差 末点与 goal 的位置 $\sqrt{\Delta x^2+\Delta y^2}$、航向 $\lvert\Delta\psi\rvert$ 参考 ≤0.5m / ≤5°

半径口径：==验收标准按真车 4.5m==（硬约束）；当前 max_turning_ratio=0.5 使规划搜索的最小半径 ≈7.65m，大于 4.5m，满足。实测值以 traj_eval.py 算出的 $r_{\min}$ 为准。

验收③：理想跟随到终点

simple_planning_simulator 理想跟随含倒车段的泊车轨迹，车从起点运动并停在车位终点。验收①的前进 / 后退入库动图（见上）即跟随过程的直接证据——车按规划轨迹完成侧方入库并停住；末点是否落在车位终点、是否停稳，以验收②的末点停位误差数据为据（参考 ≤0.5m / ≤5°）。

RS 曲线在规划框架中的定位与功能

汽车不能像机器人那样原地旋转，必须遵守非完整运动学约束，即受限于最小转弯半径。

  • RS功能：在最小转弯半径约束下、允许前进 + 倒车，连接两个位姿 (x, y, θ) 的最短路径**。是 Dubins 曲线推广。
  • 为什么泊车要用它：侧方/倒车入库必须倒车，普通几何曲线（只前进）做不到；RS 给出”前进段 + 倒车段”拼接的最短解。
  • 在这个模块的作用：RS 是 freespace_planning_algorithms 算法库里的几何工具，RS曲线并非直接求解从起点到终点的最终路径，而是作为图搜索算法（A* RRT*）的节点拓展工具。主要负责两个功能：
    1. 启发函数（heuristic）：估”从当前点到终点还多远”，引导 A* 往终点搜（astar_search.cpp:227）。
    2. shoot / 终结：搜到某点时用 RS 直奔终点，命中即接上**（Day3 详读）**。

在 Hybrid A* 中，RS 曲线常被用作 Shot Heuristic (解析解剪枝)：当算法探索到一个靠近终点的节点时，如果不考虑障碍物，直接用 RS 曲线连向终点。如果这条 RS 曲线在 Costmap 上不碰撞，搜索就直接宣告成功，极大地加速了计算。

Hybrid A*（混合 A*）

为什么要有 Hybrid A*

普通 A* 的路径长这样：在栅格中心之间走折线、走 45°。一辆真实的车（阿克曼转向，有最小转弯半径）根本走不出来。如果直接拿 A* 路径给控制器，要么跟踪不上，要么得做大量后处理（平滑、拟合），还可能拟合失败。

Hybrid A* 的目标：直接在车辆的”位姿空间”里搜索，让搜出来的路径本身就能被车开出来。

核心改动：状态从离散栅格 → 连续位姿

A* Hybrid A*
状态 栅格 (i, j) 连续位姿 (x, y, θ)，θ 是车头朝向
邻居生成 上下左右/对角 用车辆运动学前进一步（运动基元 motion primitive）
路径形状 栅格折线 真实可行驶轨迹（圆弧/RS 曲线段拼接）

连续空间怎么搜索？——3D 栅格去重

纯连续位姿有无限多个，没法搜。Hybrid A* 的巧思：

  • 把连续位姿 (x, y, θ) 投影到一个 3D 离散栅格（x 分箱、y 分箱、θ 分箱）。
  • 每个 3D cell 里只保留一个状态（通常是代价最低的那个，或第一个到达的）。
  • 这样搜索空间是有限的（分辨率完备），但每个状态本身又是连续的真实位姿——路径连续平滑，但搜索规模可控。

一句话：用离散栅格做”去重/记忆”，用连续状态做”路径生成”。这就是 “Hybrid（混合）” 二字的含义——离散 + 连续的混合。

启发函数要升级

普通欧氏距离在 Hybrid A* 里太弱（因为还要考虑朝向和不能横移），扩展会爆炸。常用更强的启发：

  1. 2D A* / Dijkstra 距离：先在 2D（忽略 θ、允许任意方向移动）上跑一遍，拿到每个格子到目标的距离，作为 h。这已经比欧氏强很多。
  2. Reeds-Shepp 距离：直接算”如果没障碍，从当前位姿到目标的最短 RS 曲线长度”。更强，但要配合碰撞检查裁剪。

工程上常把两者结合，甚至取多种启发的最大值，兼顾引导力和可采纳性。

Reeds-Shepp 加速：这是 “A* RS” 的 RS 部分

光靠运动基元一格一格扩，最后可能永远”差一点点”对不上目标的精确朝向 θ。解决办法：

  • 每次扩展出一个节点后，尝试用 Reeds-Shepp 曲线直接连到目标。
  • 如果这条 RS 曲线无碰撞，那就直接接上、结束搜索。
  • 这让算法能精确收敛到目标位姿，大幅减少靠近目标时的无谓扩展。

Reeds-Shepp 曲线：给定两个位姿，在”车能前进、能后退”的前提下，存在解析的最短路径（由若干段圆弧+直线组合，公式可查表）。能前进能后退 = 适合泊车这种需要揉库的场景。
Dubins 曲线：只能前进不能后退的对应物，适合单向行驶。

Hybrid A* 搜索逻辑

状态空间：(x, y, θ) 三维网格离散

连续的位姿 (x, y, θ) 没法直接搜索，所以 setMap（L138）按 costmap 分辨率把 x, y 切成网格，再按 theta_size（默认 48）把航向 θ 切成 48 档，得到一个 3D 数组 nodes_[y][x][theta]。每个格子放一个 AstarNode。任意连续位姿都被 pose2index 吸附到最近的格子——这就是 Hybrid A* 控制状态数量、避免无限细分的关键。

代价的 gc 是连续累积的（不量化），但节点身份是按格子去重的：同一个格子只算一个节点（status 三态），所以不会无限扩展。

运动基元表：用对称性一次生成

createTransitionTable（L76）离线把所有合法的”走一步”枚举出来：

基元 生成位置 含义
前进直行 forward_straight（L95） 走 step_min = R_min·dtheta 直线
前进左弧 forward_left（L100） 半径 R 左转 dtheta
前进右弧 forward_left.flipped()（L102） 镜像得右转
倒车各向 .reversed()（L116） 前进各基元取反，得倒车版本

半径从 R_min 到 R_max 分 turning_radius_size+1 档（默认 12 档），每档有左、右两条弧，再加一条直行。每个 θ 档把这些基元用 rotated(theta) 转到当前朝向下（L107–112）；use_back=true 时再追加 reversed() 的倒车基元（L114–118）。

所有基元的弧半径都 ≥ R_min，所以每条转移边天然满足最小转弯半径约束——这是 Hybrid A* 路径”车能跟”的根本保证。

‘节点拓展演示图’

代价与启发

  • gc：从起点到当前节点的累积代价，每步加 move_cost。
  • move_cost：正常就是基元的 distance；如果这一步换了方向（is_back 翻转，即前进↔倒车切换），乘 reverse_weight（默认 2.0，L265）——这就是换向惩罚，让算法优先少换挡。
  • hc：estimateCost 给的 RS 距离 × distance_heuristic_weight。
  • cost = gc + hc，优先队列每次取最小。

reverse_weight 的存在，解释了为什么最终轨迹的”倒车段”是分段、不频繁的：换一次方向要付 2 倍代价，算法会尽量凑成”前进一长段 + 倒车一长段”，而不是反复进退。

终点判定与回溯

isGoal（L336）不是”恰好等于终点”，而是检查节点是否落进终点周围的范围盒：横向 < lateral_goal_range、纵向 < longitudinal_goal_range、航向差 < angle_goal_range（都除以 2 作半宽）。落进盒子就算到达。only_behind_solutions 还能要求解必须从终点后方进入。

命中后 setPath（L300）从终点节点沿 parent 指针一路回溯到起点，收集每个节点的位姿和 is_back，最后 std::reverse（L328）反转成起点→终点顺序。这里的 is_back 字段会一路传到 Day1 的节点层：createTrajectory 根据它给速度取正负号，getReversingIndices 再标出倒车切换点——倒车段的”源头”就在这一步。

为什么需要局部避障

Day1–3 产出的轨迹是基于规划时刻的地图算出来的。车开起来后环境是动态的：行人可能走进轨迹、前车可能减速、点云里突然冒出一根柱子。如果死守原轨迹，迟早撞。所以规划栈在 freespace_planner 之后挂了两层运行时兜底，分工按”速度区间”切：

obstacle_cruise_planner ==行驶中== 轨迹前方的障碍 撞得上 → 插停车点 / 限速跟车

一个管”别乱起步”，一个管”开起来别撞”。下面分开走读。

obstacle_cruise_planner：前方障碍 → 停车 或 减速跟车

定位与两层规划

行驶中，轨迹前方有障碍时由它处理。它把障碍分成两类，分别走两条规划路径：

  • 要停的：障碍挡死，必须在障碍前插一个零速点 → generateStopTrajectory
  • 要跟的：障碍在动（前车），算一个安全跟车限速，减速跟着 → generateCruiseTrajectory

PlannerInterface 是基类（planner_interface.hpp L41），巡航部分是纯虚函数 generateCruiseTrajectory（L72），有两个实现：OptimizationBasedPlanner（node.cpp L351，默认，用优化解速度）和 PIDBasedPlanner（L354，用 PID）。停车逻辑 generateStopTrajectory 写在基类里（L69），两种 planner 共用。

整条触发链在 onTrajectory（node.cpp L533）里串起来：

障碍怎么变成 target： getTargetObstacles（L662）对每个感知物体，用 polygon_utils 算它的多边形和自车轨迹的交点（collision_points）。只有==轨迹真的会穿过障碍==（有 collision_points）的物体才进 target_obstacles。这一步是整个触发的总开关：没交点 = 撞不上 = 直接忽略。

停车触发：generateStopTrajectory（planner_interface.cpp L86）

从 target_obstacles 里挑最近的停车障碍，算到碰撞点的弧长距离，减掉安全 margin 和车头偏移，得到该停的位置，再用 insertStopPoint（L196）把一个零速点插进轨迹。

里面有个关键的可行性检查 calcMinimumDistanceToStop（L75）——就是高中刹车距离公式：

$$d_{brake} = \frac{v_0^2}{2,|a_{min}|}$$

含义：以当前车速 $v_0$、用极限制动 $a_{min}$，最少要多少距离才能停下。如果障碍比这个距离还近（closest_obstacle_stop_dist < feasible_stop_dist，L186），说明==按极限制动也停不住==，会撞，于是置 will_collide_with_obstacle = true 发告警（L184-189）。这是个兜底的安全诊断，不是触发条件本身。

巡航触发：generateCruiseTrajectory + RSS 距离

对于要”跟”的障碍（前车），用 RSS（Responsibility-Sensitive Safety）模型算安全跟车距离，定义在 calcRSSDistance（planner_interface.hpp L127）：

$$d_{rss} = v_{ego}\cdot t_{idle} + \frac{v_{ego}^2}{2,|a_{ego}|} - \frac{v_{obj}^2}{2,|a_{obj}|} + margin$$

四项分别是：自车反应时间走的距离 + 自车制动距离 − 障碍自己的制动距离 + 余量。第三项减掉是因为前车也在减速，不需要给它留”完全停下”的整段距离。算出来的 $d_{rss}$ 用来定限速：跟车距离不够就降速，够就维持。输出是 VelocityLimit（限速指令），不是急停——和停车触发的区别就在这。

话题通信基于发布订阅模式。

话题通信是一种以发布订阅的方式实现不同节点之间数据传输的通信模型。数据发布对象称为发布方，数据订阅对象称之为订阅方，发布方和订阅方通过话题相关联，发布方将消息发布在话题上，订阅方则从该话题订阅消息，消息的流向是单向的。
